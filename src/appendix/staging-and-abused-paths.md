# Staging & abused locations (Win ↔ Linux ↔ macOS)

Where payloads land before they run, where collected data sits before it leaves, and
which directories are writable by an unprivileged process *by default*. These are the
filesystem chokepoints behind Execution, Persistence, and Collection: an attacker has to
put a file somewhere, and the somewhere is rarely random. It is the path that is both
writable without elevation and innocuous enough to escape scrutiny. This appendix is the
path-level companion to the behavior chapters, the concrete `dir=` you point a sensor at.

```admonish note title="Methods & confidence"
Sources: vendor threat reporting (Microsoft, Trend Micro, SentinelOne, ESET, Picus,
Red Canary, Sandfly, Cado, Sysdig/Aqua, Objective-See, CISA/FBI advisories, NCC/DFIR
research) + MITRE ATT&CK (technique/software/group mappings) + OS vendor file-system
docs, as-of **June 2026**. ATT&CK mappings are authoritative; threat-to-path attributions
are cited per row. **Default writability is version- and edition-specific**, Windows ACLs
below were read live on a Win11 Home build 26200 host on 2026-06-19, Linux modes on
Ubuntu 24.04 / Debian 12 / Fedora 41 via `ls -ld`, both where marked *verified*; treat
those as that build's state and re-check the box in front of you. Claims that could not be
confirmed on a live host are prefixed `unverified:`.
```

## Behaviors → threats → staging paths

lives. Mechanism diverges across OS; the *role* (drop-and-run sink, persistence-stable
home, in-memory non-path) is the invariant.

## Windows

ACL rows marked *verified* were read with `icacls`/SDDL on a Win11 Home build 26200 host
on 2026-06-19. The recurring Windows pattern: the abused locations are not loose because
of a bug, they are **create-new-content writable by design** (you own what you create),
container-inherit, with read+execute on existing items. The abuse is dropping *your own*
file, not tampering with system files.

### Exec / script staging

### World-writable / loose-ACL dirs

Windows has no `chmod 777` equivalent, but a handful of locations are writable by every
standard user by default. The honest framing (verified rows): these are **create-new**
writable with read+execute on existing content, drop your own file, do not tamper with
system files.

| Path | Why writable by default / verified state | ATT&CK | Named threats (cited) | Detection note |
|---|---|---|---|---|
| `C:\ProgramData` (`%ProgramData%`) | Closest Windows analog to world-writable, **by design**. Verified (Win11 Home 26200, 2026-06-19): `O:SYG:SY…(A;OICI;0x1200a9;;;BU)(A;CI;DCLCRPCR;;;BU)`, Users get read+execute everywhere PLUS an inheritable container ACE to create files/subdirs. Sanctioned drop for machine-wide all-user app state, so every standard user must create per-app data without admin rights. Create-new content (you own what you create), container-inherit, read+execute on existing, not a tamper-everything hole. | T1222.001 (permissive default is the enabler) → T1059/T1204/T1036/T1074 | DarkGate, AutoIt loader staged/executed (Trend Micro, Oct 2023); IcedID, artifacts in ProgramData (MITRE S0483) | Sysmon EID 11 on first-level subfolder create by non-installer |
| `C:\Users\Public` (`%PUBLIC%`) and child media/Downloads folders | Verified loose (this host, 2026-06-19): `INTERACTIVE`/`SERVICE`/`BATCH` granted Modify + DeleteSubdirectoriesAndFiles, inherited (`IU/SU 0x1301ff OICIIO` + `0x1200af`). The Public profile exists so any logged-on user can SHARE files with any other local user → create/modify/delete by INTERACTIVE is intentional. The most genuinely everyone-read+write+delete+execute location an unprivileged attacker has → perennial cross-user dead-drop. | T1222.001 → T1059/T1204/T1074.001/T1036 | Gamaredon, `%PUBLIC%` subfolder staging (MITRE G0047); PlugX, components in Public (MITRE S0013) | Sysmon EID 11 + EID 1 with `Image`/`TargetFilename` under `C:\Users\Public\` |
| `C:\Windows\Temp` | `unverified:` live ACL (icacls denied unelevated; UAC not driveable non-interactively). Microsoft-documented default gives Users a restricted write (create files/dirs, limited read/traverse of others') so multiple service/SYSTEM contexts share a temp without exposing each other's data. Abused because it is `%SystemRoot%`-resident ("trusted") and the default working/temp dir for service-launched + lateral tooling. Re-check the exact ACE set per build. | T1222.001 / T1569.002 working-dir abuse / T1059 | PsExec lateral movement service artifacts (MITRE S0029; T1569.002); Conti deployment tooling (CISA AA21-265A) | Sysmon EID 11 + SYSTEM/service-context writer |
| `C:\PerfLogs` | `unverified:` live ACL. Documented default ~ Users read+execute with constrained write; created for PerfMon data-collector output. Abuse value is its **emptiness** plus a system-looking name, not a uniquely loose ACL, the lowest-noise drop dir on a stock box. | T1036 (masquerade as system path) / T1059 / T1074.001 | Hive, `.zip` staging in `C:\PerfLogs` (NCC Group deep-dive; the CISA advisory AA22-321A does not mention PerfLogs) | Sysmon EID 11, alert on *any* create given baseline emptiness |
| Misconfigured third-party app dirs (non-default writable `C:\<vendor>` / weak Program Files subfolder) | Not a Windows default, a vendor/installer misconfiguration leaving a Users-writable dir, often in a service/SYSTEM binary's DLL search path. Enables planting + search-order hijack from a standard-user write. | T1574.001 / T1574.002 / T1222.001 (weak ACL enabler) | PlugX, side-load via writable app dir (MITRE S0013); Lazarus / many APTs (MITRE T1574.002 procedure examples) | Config-audit detection, not an event rule, periodic `accesschk -w` / `Get-Acl` sweep of service binary search paths; alert on new user-writable entries + Sysmon EID 11 (DLL write) + EID 7 (unsigned DLL load by signed image) |

## Linux

Default-writable locations are the POSIX sticky-bit world-writable dirs (`1777`) plus the
user's own `$HOME`. The recurring pattern differs from Windows: Linux abuse leans on the
*loader* (`LD_PRELOAD` / `ld.so.preload`), on `/tmp`-family world-writable dirs, and on
**fileless** primitives, and, critically, much of cloud Linux runs `auditd` **off by
default**, so these writes are often unobserved rather than absent (see
[safeguard-pressure](safeguard-pressure.md)). Modes below verified on Ubuntu 24.04 /
Debian 12 / Fedora 41 via `ls -ld`, June 2026, where marked *verified*.

### Exec / script staging

| Path | Why abused / default state | ATT&CK | Named threats (cited) | Detection note |
|---|---|---|---|---|
| `/tmp` | **Verified `1777`** (`drwxrwxrwt`: world-writable + sticky) on all three distros. The single most common Linux drop-and-run sink: droppers `curl`/`wget` a stage-2, `chmod +x`, exec from `/tmp`, no priv-esc, any compromised account works. The sticky bit only stops *other* users deleting your file; it does not stop creation or execution. `unverified:` `noexec` is NOT the default on stock Ubuntu/Debian root-fs `/tmp`, so exec-from-`/tmp` generally works out of the box; some hardened hosts mount `/tmp` `tmpfs,noexec`. | T1059.004 (Unix Shell) / T1204 / T1105 (Ingress Tool Transfer) / T1036.005 | Kinsing/Kdevtmpfsi, downloads `kdevtmpfsi`+`kinsing` to `/tmp`, `chmod 777`, executes after exploiting exposed Docker/cloud apps (Trend Micro / Aqua, 2021-2024; MITRE S0599); BPFDoor, copies to `/tmp` under masqueraded names (e.g. `/tmp/zabbix_agent`) and runs (Elastic / Sandfly / Trend Micro, 2022-2025; MITRE S1161); TeamTNT, stages tooling + cred-scraping payloads in `/tmp` during cloud/Docker compromise (Cado/Aqua, 2020-2023) | auditd `-w /tmp -p wa` / `-F dir=/tmp`; bpftrace `file-modify.bt` on `vfs_write` under `/tmp`; gate on exec from `/tmp` (`-S execve -F dir=/tmp`), write→chmod+x→exec of the same inode is the invariant |
| `/dev/shm` | **Verified `1777` `tmpfs`** (`drwxrwxrwt`) on all three distros. RAM-backed, world-writable → prime **fileless** staging: payload lives only in memory, vanishes on reboot, defeats disk-forensics and on-disk FIM baselines. `unverified:` whether a given host enforces `noexec`, container runtimes (Docker default observed here) and some hardened hosts mount `/dev/shm noexec,nosuid,nodev`, blocking direct exec; attackers then pivot to `memfd_create` or an interpreter (`python`/`perl`) reading from shm. | T1059.004 / T1105 / T1564 (in-RAM concealment, no sub-technique) | BPFDoor **v1**, working dir under `/dev/shm` (e.g. `/dev/shm/kdmtmpflush`), deleted itself from disk; **v2 dropped this** for pure memory residency via BPF magic-packet activation (Sandfly Security, BPFDoor analysis, 2023-2025); Commando Cat, container-escape crypto-jacking explicitly drops+executes in `/dev/shm` instead of `/tmp` to evade disk scanning (Cado Security, 2024) | auditd `-F dir=/dev/shm -p wa`; bpftrace on `vfs_write` to `/dev/shm`; an `execve` whose path is under `/dev/shm` is high-signal with little baselining, treat near-allowlist-only |
| `/var/tmp` | **Verified `1777`** (`drwxrwxrwt`) on all three distros. ## macOS

macOS world-writable dirs mirror Linux (`/tmp`, `/private/var/tmp`), but the dominant
staging story is the user's `~/Library`, writable without privilege and the home of
LaunchAgents, and the **`com.apple.quarantine` attribute gap**, where SIP/Gatekeeper
displacement pushes adversaries (see [safeguard-pressure](safeguard-pressure.md)). `/tmp`
is a symlink to `/private/tmp`. Sensor mapping uses the guide's macOS tiers: ESF (EDR
tier, primary) via `eslogger`/Mac Monitor, `NOTIFY_CREATE`/`WRITE`/`CLOSE` see the drop,
`NOTIFY_EXEC` + inline code-signing fields (team ID, signing ID, cdhash,
`is_platform_binary`) catch the run; the unified log has **no reliable exec event**, so
without an ESF agent you are effectively blind to execution from any of these paths.

```admonish tip title="Detection: gating without drowning"
Every path above is full of legitimate use, `%TEMP%`, `/tmp`, `/dev/shm`, `~/Library`,
and `~/Library/Caches` see thousands of benign writes an hour. A bare "alert on write to
`/tmp`" detection is unworkable. Gate on **path + at least one anomaly axis**:

- **Anomalous parent lineage**, Office/browser/archiver → EXE in `%TEMP%`; a
  network-facing/non-interactive service (`www-data`, `nginx`, `php-fpm`, an SSH one-liner)
  → write+exec in `/tmp`; a launchd/Terminal/`osascript` ancestor for an exec from a macOS
  staging dir; a non-installer process creating a top-level `C:\ProgramData\` folder.
- **Unsigned / unexpected writer**, a write to `…\spool\drivers\color`, a
  `~/Library/LaunchAgents` plist, or a `~/.zshenv` line authored by an unsigned or
  unexpected binary (macOS ESF carries the signer inline, score "non-platform,
  unsigned/ad-hoc Mach-O executing FROM a staging dir" in one event).
- **Rarity**, `C:\PerfLogs` and the root of `C:\ProgramData` are normally empty/stable;
  `/etc/ld.so.preload` and `/run/lock` executables are normally absent; `/Users/Shared`
  writes are human-paced, *any* anomalous write is high-signal. Baseline first.
- **File type**, a PE/ELF/Mach-O or archive (`.zip`/`.7z`/`.rar`/`.tar.gz`, split
  `.partNN`/`.r00`/`.z01`) where data files normally do not appear (ProgramData/Public bulk
  archives = exfil prep). For exfil, the invariant is **archive-create-then-egress**: tie
  the archive write to an outbound connection from the same process tree within minutes.
- **The write→chmod+x→exec chain** (Unix), a write into a world-writable dir AND a
  `chmod +x` AND a subsequent `execve` of that inode is the chokepoint cut; legit temp
  files are rarely made executable and run. `memfd_create` has no path, anchor instead on
  `/proc/<pid>/exe` → `/memfd:…(deleted)`.

Name the sensor per OS so the detection is buildable:

| OS | Sensor for a write/drop in these locations |
|---|---|
| **Windows** | Sysmon **EID 11** (FileCreate), primary; **EID 15** (FileCreateStreamHash) for ADS; EID 23/26 for self-cleanup; pair with EID 1 (parent lineage), EID 7 (ImageLoad, side-load), EID 13 (Run-key), EID 3 (egress) |
| **Linux** | `auditd -F dir=<path> -p wa` (or `-w <path> -p wa`); bpftrace `file-modify.bt` on `vfs_write`; exec gate `-S execve,execveat -F dir=<path>`; `-S chmod,fchmod,fchmodat` for the `+x` step; eBPF/Tetragon `process_exec` for memfd-backed images |
| **macOS** | ESF `ES_EVENT_TYPE_NOTIFY_CREATE` / `NOTIFY_WRITE` (via `eslogger` / Mac Monitor), `NOTIFY_EXEC` with inline signer/`is_platform_binary`, `NOTIFY_SETEXTATTR` for `xattr -d com.apple.quarantine`, `NOTIFY_SETMODE` for `chmod +x`, macOS has no `auditd`-style `dir=` watch; ESF is the equivalent |

The SIEM tier is weaker than the EDR tier in every column, `auditd` is off by default on
much of cloud Linux, and the macOS unified log has no clean file-create or exec event. An
empty cell here may be *unobserved*, not *absent*. See
[telemetry cheat-sheet](telemetry-cheatsheet.md) for the EDR-vs-SIEM gap per sensor,
[safeguard-pressure](safeguard-pressure.md) for why a staging path is hot on one OS and
cold on another, and the [threat-coverage-matrix](threat-coverage-matrix.md) for which
threats populate each column.
```
